Pinegrow CMS Demo

This is a template instance page; it is indicated as an instance with the HTML tag <html lang="en" data-pgc-master="master.html">. The value of the data-pgc-master attribute is the filename of this page's master template (in this case, master.html).

http://docs.pinegrow.com/master-pages/#use-a-master-page-in-a-project gives instructions for creating instance pages from a master page. This process may result in Pinegrow creating a screenshots directory with a thumbnail of the master template.

This page has an editable area definition for Pinegrow's CMS mode, indicated with the HTML tag <div data-pgc-field="page_content">. The value of the data-pgc-field attribute is an ID, which distinguishes it from other editable areas. As noted in http://docs.pinegrow.com/master-pages/, the master page's content is applied to all of its instances, except for editable areas.

Besides master/instance pages, I have also looked at using partial pages as noted in http://docs.pinegrow.com/working-with-partials/. Here are some notable pros and cons of partials:

  • Pros
    • Avoids duplication in the source repository (space-efficient).
    • Easier for contributors to make edits and compare revisions.
    • Can also be easily used in non-static sites.
  • Cons
    • Static hosting sites such as GitHub Pages don't necessarily render out partial pages.
      • According to GitHub's documentation at  https://help.github.com/articles/using-a-static-site-generator-other-than-jekyll/, static sites built without Jekyll should be built locally, with the output (meaning full HTML pages) pushed to the appropriate branch.
      • I could choose to edit this page as a partial within Pinegrow; however, when viewed on GitHub Pages, it would be missing the elements from the master template.
      • Another possible way to render partial pages is through a client-side JavaScript framework; however, this method is currently outside the scope of my project.
